# Лабораторная работа — Паттерн «Strategy» (Стратегия)
## Предметная область: Battle Arena

### Описание предметной области
Проект представляет собой симулятор сражений «стенка на стенку». Система позволяет двум игрокам покупать и расставлять юнитов различных классов на поле боя. После завершения фазы расстановки юниты переходят под управление ИИ, который определяет их поведение в зависимости от их роли.

### Описание проблемы (без паттерна)

В первоначальной реализации логика поведения всех типов юнитов была сосредоточена в одном методе `update` класса `Unit`. Это привело к созданию «Божественного объекта» (God Object), который знает слишком много о деталях реализации каждого подтипа.

```python
# Пример "спагетти-кода" до применения паттерна
def update(self):
    if self.unit_type == 'melee':
        # Логика ближнего боя
    elif self.unit_type == 'ranged':
        # Логика стрельбы и отхода
    elif self.unit_type == 'healer':
        # Поиск союзника для лечения
    # ... и тд.
```

**Проблемы такого подхода:**
1.  **Нарушение принципа Open/Closed:** Для добавления нового типа юнита (например, «Мага») приходится редактировать основной класс `Unit`, что повышает риск сломать логику существующих персонажей.
2.  **Сложность поддержки:** Огромный блок ветвлений (`if-elif-else`) становится нечитаемым при увеличении количества ролей.
3.  **Невозможность смены поведения:** Трудно изменить логику юнита «на лету» (например, если раненый воин должен начать убегать).

---

### Решение: применение паттерна «Strategy»

Паттерн «Стратегия» позволил вынести алгоритмы поведения в отдельную иерархию классов. Теперь класс `Unit` не содержит конкретных инструкций по бою, а делегирует их выполнение объекту-стратегии.

#### 1. Интерфейс и абстракция
Был создан базовый класс `IBehavior`, использующий модуль `abc` (Abstract Base Classes) для определения жесткого контракта:

```python
class IBehavior(ABC):
    @abstractmethod
    def update(self, unit, target): pass
```

#### 2. Конкретные реализации
Для каждой роли создан свой класс. Это позволило изолировать сложную логику. Например, в `AssassinBehavior` реализован алгоритм ускорения при сближении, а в `HealerBehavior` — приоритет поиска союзника с минимальным HP:

```python
class HealerBehavior(IBehavior):
    def update(self, unit, target):
        # Логика поиска раненого союзника
        # Логика лечения или атаки, если союзников нет
```

#### 3. Связывание в Контексте
Класс `Unit` при инициализации выбирает нужную стратегию через словарь `BEHAVIOR_MAP` и использует её в каждом кадре:

```python
class Unit(Entity):
    def __init__(self, ...):
        # Динамическое создание объекта стратегии
        self.behavior = self.BEHAVIOR_MAP.get(behavior_type, MeleeBehavior)()

    def update(self):
        # Делегирование выполнения алгоритма
        if target:
            self.behavior.update(self, target)
```

### Вывод: преимущества применения паттерна

| Преимущество | Результат в коде |
|--------------|----------------------|
| **Масштабируемость** | Новые типы юнитов добавляются через создание новых классов стратегий без изменения `Unit.py`. |
| **Чистота кода** | Основной цикл обновления юнита сократился до вызова одного метода `self.behavior.update()`. |
| **Инкапсуляция** | Детали того, как именно лучник «кайтит» врага, скрыты внутри `RangedBehavior`. |
| **Удобство отладки** | Для редактирования движений убийцы не нужно перелопачивать спагетти, так как код изолирован в соответствующем классе. |
