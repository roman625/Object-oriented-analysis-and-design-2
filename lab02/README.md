
# Лабораторная работа — Паттерн «Composite» (Компоновщик)
## Предметная область: Система управления структурой организации

### Описание предметной области
Приложение предназначено для визуализации и управления иерархической структурой компании. Система позволяет работать с организацией как с единым целым, независимо от того, состоит ли она из отдельных сотрудников или разветвленной сети департаментов.

| Критерий | Описание | Пример значений |
|----------|----------|----------------|
| **Тип узла** | Роль в иерархии | Сотрудник (Leaf), Отдел (Composite) |
| **Уровень вложенности** | Глубина в дереве | 0 (Компания), 1 (Департамент), 2 (Группа) |
| **Агрегация данных** | Расчет показателей | Сумма зарплат всех подчиненных |

**Доступные сущности:**

| Сущность | Роль в паттерне | Функционал | Уникальные свойства |
|-----------|-----------|-----------------|-----------|
| **Сотрудник** | Лист (Leaf) | Возвращает свои данные | Должность, Навык |
| **Отдел** | Компоновщик | Хранит список `Employee` | Может содержать и людей, и отделы |

---

### Описание проблемы (без паттерна)

В реализации без паттерна пришлось бы разделять логику обработки отдельного человека и целого отдела. Это привело бы к перегрузке кода проверками типов (`if-else` или `switch`) при каждом обходе дерева:

```cpp
// Плохой пример: высокая связность и ручной перебор типов
void PrintAll(void* entity, bool isDept) {
    if (isDept) {
        Department* d = static_cast<Department*>(entity);
        for (auto sub : d->subDepts) { // Нужно знать структуру векторов
             // Снова нужно проверять: это под-отдел или человек?
        }
    } else {
        Person* p = static_cast<Person*>(entity);
        std::cout << p->name;
    }
}
```

**Проблемы такого подхода:**

| Проблема | Последствия |
|----------|-------------|
| **Нарушение OCP** | При добавлении нового типа (например, «Филиал») нужно менять все функции поиска и расчета бюджета. |
| **Сложность клиента** | GUI должен знать, как работать с разными классами отдельно, что раздувает код обработчиков кнопок. |
| **Дублирование** | Логика рекурсивного обхода пишется заново для каждой операции (печать, поиск, бюджет). |

### Решение: применение паттерна «Composite»

#### 1. Компонент (Общий интерфейс)
Абстрактный класс `Employee` задает общий контракт для листьев и контейнеров:

```cpp
using EmployeePtr = std::shared_ptr<class Employee>;

class Employee {
public:
    virtual double getSalary() const = 0; 
    virtual void add(EmployeePtr employee) = 0; 
    virtual void printStructure(ostringstream& oss, int indent = 0) const = 0;
    virtual void printBudgetReport(ostringstream& oss, int indent = 0) const = 0;
};
```

#### 2. Лист: Сотрудник (Person)
Реализует базовое поведение, не имея вложенных элементов. Использует `shared_from_this` для безопасной передачи указателей:

```cpp
class Person : public Employee, public std::enable_shared_from_this<Person> {
    double getSalary() const override { return salary_; }
    void add(EmployeePtr) override { /* У листа нет детей */ }
    void printStructure(ostringstream& oss, int indent) const override {
        oss << string(indent * 2, ' ') << "Сотрудник: " << name_ << "\r\n";
    }
};
```

#### 3. Компоновщик: Отдел (Department)
Хранит единый список `EmployeeList` и делегирует выполнение операций своим элементам:

```cpp
class Department : public Employee, public std::enable_shared_from_this<Department> {
    EmployeeList employees_; // Хранит ЛЮБЫХ Employee (и людей, и отделы)

    double getSalary() const override {
        double total = 0;
        for (const auto& e : employees_) total += e->getSalary(); // Полиморфный вызов
        return total;
    }
    void add(EmployeePtr e) override { if (e) employees_.push_back(e); }
};
```

#### 4. Клиент (GUI Logic)
Работает с корневым объектом `company`, не вникая в его внутреннее устройство:

```cpp
// Вызов одной команды запускает рекурсивную цепную реакцию
case 2: // Кнопка "Бюджет"
    company->printBudgetReport(oss); 
    // Нам не важно, сколько там уровней вложенности — ответ придет сам.
```

### Сравнительная таблица структурных паттернов

| Паттерн | Основная цель | Ключевая особенность | Отношение к интерфейсу |
| :--- | :--- | :--- | :--- |
| **Компоновщик** (Composite) | Организация иерархии «часть-целое». | Позволяет работать с деревом как с единым объектом. | **Сохраняет** общий интерфейс для группы и листа. |
| **Декоратор** (Decorator) | Динамическое расширение функционала. | Оборачивает объект («луковица»), добавляя слои логики. | **Сохраняет** интерфейс исходного объекта. |
| **Мост** (Bridge) | Разделение абстракции и реализации. | Позволяет менять логику и платформу независимо. | **Разделяет** один класс на две иерархии. |
| **Фасад** (Facade) | Упрощение работы со сложной системой. | Предоставляет единую «точку входа» (пульт управления). | **Создает новый** интерфейс. |
| **Заместитель** (Proxy) | Контроль доступа или ленивая загрузка. | Суррогат реального объекта, перехватывающий вызовы. | **Сохраняет** интерфейс реального объекта. |

### Вывод: преимущества применения Composite

| Преимущество | Как реализовано в коде |
|--------------|----------------------|
| **Принцип OCP** | Можно добавить класс `Freelancer`, и функции бюджета/поиска продолжат работать без изменений. |
| **Упрощение клиента** | Обработчик `WM_COMMAND` вызывает один метод у `company`, не строя циклы обхода. |
| **Рекурсивная мощь** | Расчет бюджета всей компании — это одна строка кода, инициирующая обход всего дерева. |
| **Гибкость** | В методе `handleDelete` штат удаляемого отдела легко переносится в другой отдел через `add()`. |
