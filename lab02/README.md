# Лабораторная работа — Паттерн «Composite» (Компоновщик)
## Предметная область: Система управления структурой организации

### Описание предметной области
Приложение **OrgSystem** предназначено для визуализации и управления иерархической структурой компании. Система позволяет работать с организацией как с единым целым, независимо от того, состоит ли она из отдельных сотрудников или разветвленной сети департаментов.

| Критерий | Описание | Пример значений |
|----------|----------|----------------|
| **Тип узла** | Роль в иерархии | Сотрудник (Leaf), Отдел (Composite) |
| **Уровень вложенности** | Глубина в дереве | 0 (Компания), 1 (Департамент), 2 (Группа) |
| **Агрегация данных** | Расчет показателей | Сумма зарплат всех подчиненных |

**Доступные сущности:**

| Сущность | Роль в паттерне | Функционал | Уникальные свойства |
|-----------|-----------|-----------------|-----------|
| **Сотрудник** | Лист (Leaf) | Возвращает свои данные | Должность, Навык |
| **Отдел** | Компоновщик | Хранит список `Employee` | Может содержать и людей, и отделы |

---

### Описание проблемы (без паттерна)

В реализации без паттерна пришлось бы разделять логику обработки отдельного человека и целого отдела. Это привело бы к перегрузке кода проверками типов (`if-else` или `switch`) при каждом обходе дерева:

```cpp
// Плохой пример: высокая связность и ручной перебор типов
void PrintAll(void* entity, bool isDept) {
    if (isDept) {
        Department* d = static_cast<Department*>(entity);
        for (auto sub : d->getSubs()) {
            // Снова нужно проверять: это под-отдел или человек?
        }
    } else {
        Person* p = static_cast<Person*>(entity);
        std::cout << p->getName();
    }
}
```

**Проблемы такого подхода:**

| Проблема | Последствия |
|----------|-------------|
| **Нарушение OCP** | При добавлении нового типа (например, «Филиал») нужно менять все функции поиска и расчета бюджета. |
| **Сложность клиента** | GUI должен знать, как работать с разными классами отдельно, что раздувает код обработчиков кнопок. |
| **Дублирование** | Логика рекурсивного обхода пишется заново для каждой операции (печать, поиск, бюджет). |

---

### Решение: применение паттерна «Composite»

#### 1. Компонент (Общий интерфейс)
Абстрактный класс `Employee` задает общий контракт:

```cpp
class Employee {
public:
    virtual double getSalary() const = 0; // Единый метод для всех
    virtual void add(shared_ptr<Employee> e) = 0; // Управление детьми
    virtual void printStructure(ostringstream& oss, int indent = 0) const = 0;
};
```

#### 2. Лист: Сотрудник (Person)
Реализует базовое поведение, не имея вложенных элементов:

```cpp
class Person : public Employee {
    double getSalary() const override { return salary_; }
    void add(shared_ptr<Employee>) override { /* Пусто: у листа нет детей */ }
    void printStructure(ostringstream& oss, int indent) const override {
        oss << string(indent * 2, ' ') << "Сотрудник: " << name_ << "\r\n";
    }
};
```

#### 3. Компоновщик: Отдел (Department)
Хранит список указателей на интерфейс и делегирует им работу:

```cpp
class Department : public Employee {
    vector<shared_ptr<Employee>> employees_; // Хранит ЛЮБЫХ Employee

    double getSalary() const override {
        double total = 0;
        for (const auto& e : employees_) total += e->getSalary(); // Рекурсивный вызов
        return total;
    }
    void add(shared_ptr<Employee> e) override { employees_.push_back(e); }
};
```

#### 4. Клиент (GUI Logic)
Работает с «Корнем» (Company), не зная о его внутренней структуре:

```cpp
// Вызов одной команды запускает цепную реакцию по всему дереву
case 2: // Кнопка "Бюджет"
    res = company->printBudgetReport(oss); 
    // Нам не важно, сколько там отделов и людей — ответ придет сам.
```

### Сравнительная таблица структурных паттернов



| Критерий | Composite | Decorator | Proxy | Facade | Adapter |
|----------|---------------|------------------|-----------|---------|-----------|
| **Цель** | Иерархия часть-целое | Динамическое расширение | Контроль доступа | Упрощение API | Совместимость |
| **Структура** | Дерево объектов | Слоеная обертка | Объект-заменитель | Точка входа | Переходник |
| **Интерфейс** | Общий для всех | Сохраняется | Сохраняется | Новый (простой) | Новый (нужный) |

---

### Вывод: преимущества применения Composite

| Преимущество | Как реализовано в коде |
|--------------|----------------------|
| **Принцип OCP** | Можно добавить класс `ProjectGroup`, и функции бюджета/поиска продолжат работать без изменений. |
| **Упрощение клиента** | Обработчик кнопок в Win32 вызывает один метод у `company`, не строя циклы обхода. |
| **Рекурсивная мощь** | Расчет бюджета всей компании — это одна строка кода, вызывающая сотни подрасчетов. |
| **Гибкость** | Структуру организации можно менять на лету, перекладывая «папки» (отделы) друг в друга. |

### Запуск приложения (Инструкция)

**Требования:**
- IDE: Visual Studio 2019/2022.
- Подсистема: Windows (/SUBSYSTEM:WINDOWS).

**Использование:**
1. Нажмите **«Структура»** для просмотра иерархии.
2. Нажмите **«Бюджет»** для рекурсивного финансового отчета.
3. Команда **Добавить**: `[Имя_Отдела] [Имя_Нового] [Навык] [Поз] [0]` (создаст подотдел).
