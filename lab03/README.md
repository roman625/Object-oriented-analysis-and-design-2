# Лабораторная работа — Паттерн «Strategy» (Стратегия)
## Предметная область: Система поведения юнитов в игре «Battle Arena»

### Описание предметной области
Приложение представляет собой аркадную стратегию в реальном времени, где юниты двух команд сражаются на ограниченной арене. Каждый тип юнита обладает уникальным поведением: перемещением, атакой, лечением или комбинацией действий.

| Критерий | Описание | Пример значений |
|----------|----------|----------------|
| **Тип поведения** | Алгоритм принятия решений | Ближний бой, Дистанционная атака, Лечение |
| **Изоляция логики** | Возможность тестирования | Поведение тестируется отдельно от Entity |

**Доступные сущности:**

| Сущность | Роль в паттерне | Функционал | Уникальные свойства |
|-----------|----------------|-----------------|---------------------|
| **IBehavior** | Стратегия (интерфейс) | Задает контракт `update(unit, target)` | Абстрактный метод для всех алгоритмов |
| **Melee/Ranged/etc** | Конкретная стратегия | Реализует уникальную логику боя | Параметры: скорость, урон, дистанция |
| **Unit** | Контекст (Context) | Хранит ссылку на `IBehavior`, делегирует выполнение | `self.behavior.update(self, target)` |

---

### Описание проблемы (без паттерна Strategy)

В реализации без паттерна вся логика поведения «зашивается» в один большой метод `update()` через каскад условных операторов. Это приводит к нарушению принципа единственной ответственности:

```python
# Плохой пример: монолитный метод с высокой цикломатической сложностью
def update(self):
    # ... общая логика ...
    if self.unit_type == 'melee':
        # 15 строк логики воина
    elif self.unit_type == 'ranged':
        # 20 строк логики лучника + отступление
    elif self.unit_type == 'healer':
        # 30 строк: поиск союзника + приоритеты + лечение
    # ...
```

**Проблемы такого подхода:**

| Проблема | Последствия |
|----------|-------------|
| **Нарушение Open/Closed** | Добавление нового класса (например, «Маг») требует правки существующего метода `update()`, что рискует сломать старую логику. |
| **Сложность тестирования** | Невозможно протестировать поведение «Лучника» изолированно — нужно запускать всю игру и создавать сцену. |
| **Дублирование контекста** | Метод `update()` раздувается за счёт смешивания общей логики (физика, границы) и специфичной (боевая). |

---

### Решение: применение паттерна «Strategy»

#### 1. Стратегия (Общий интерфейс)
Абстрактный класс `IBehavior` унифицирует вызов поведения. Контекст (`Unit`) не знает, какой именно алгоритм выполняется:

```python
from abc import ABC, abstractmethod

class IBehavior(ABC):
    @abstractmethod
    def update(self, unit, target): 
        """Выполняет логику поведения для данного юнита"""
        pass
```

#### 2. Конкретные стратегии: Инкапсуляция алгоритмов
Каждый класс поведения инкапсулирует только свою логику. Например, стратегия `HealerBehavior` содержит сложный приоритетный выбор цели, не засоряя основной класс:

```python
class HealerBehavior(IBehavior):
    def update(self, unit, target):
        # 1. Приоритет: поиск раненого союзника
        allies = bss.team1 if unit.team == 1 else bss.team2
        heal_target = self._find_most_injured(allies, unit.position)
        
        if heal_target:
            self._move_and_heal(unit, heal_target)
        # 2. Вторичный приоритет: атака врага
        elif target:
            self._move_and_attack(unit, target)
            
    def _find_most_injured(self, allies, pos):
        # Логика поиска, скрытая внутри стратегии
        pass
```

#### 3. Контекст: Делегирование (Unit)
Класс `Unit` становится «тонким»: он управляет состоянием, но делегирует принятие решений объекту стратегии:

```python
class Unit(Entity):
    # Словарь для динамического создания стратегий
    BEHAVIOR_MAP = {
        'melee': MeleeBehavior,
        'healer': HealerBehavior,
        # ...
    }

    def __init__(self, pos, team, behavior_type, ...):
        # ... инициализация ...
        # Полиморфное создание стратегии
        self.behavior = self.BEHAVIOR_MAP.get(behavior_type, MeleeBehavior)()

    def update(self):
        # ... общая физика и проверка границ ...
        
        # Поиск цели (контекстная логика)
        target = self._find_nearest_enemy()
        
        # ДЕЛЕГИРОВАНИЕ: вся боевая логика вынесена в стратегию
        if target:
            self.behavior.update(self, target)
```

#### 4. Клиент (BattleSystem)
Создание юнитов не зависит от сложности их поведения. Система просто указывает тип, а паттерн подставляет нужную логику:

```python
# Клиентский код не зависит от конкретных классов стратегий
u = Unit(
    pos=bss.ghost.position, 
    team=bss.current_player, 
    behavior_type=bss.selected_type  # Стратегия создастся автоматически
)
```

### Вывод: преимущества применения Strategy

| Преимущество | Как реализовано в коде |
|--------------|----------------------|
| **Изоляция сложной логики** | Логика Целителя (поиск союзников, приоритеты) полностью скрыта в `HealerBehavior`, не мешая читать код `Unit`. |
| **Соответствие Open/Closed** | Чтобы добавить класс «Маг», вы создаете новый файл `MageBehavior.py`, не касаясь ни одной строки в классе `Unit`. |
| **Упрощение тестирования** | Стратегию можно протестировать модульно, передав ей «мок-юнит» (mock object), без запуска графического движка Ursina. |
| **Гибкость Runtime** | Теоретически, поведение можно сменить на лету: `unit.behavior = EnragedBehavior()`, если юнит получил бафф «Ярость». |

### Диаграмма классов (UML)


### Когда паттерн Strategy особенно полезен?

| Ситуация | Почему Strategy выигрывает |
|----------|---------------------------|
| **Много вариаций алгоритма** | Если у вас 10+ типов юнитов, условные операторы превратят метод `update()` в нечитаемый «спагетти-код». |
| **Сложная логика** | Если поведение «Целителя» требует 50 строк кода, лучше вынести их в отдельный класс, чтобы не скроллить основной файл. |
| **Командная разработка** | Один программист может балансировать урон в `MeleeBehavior`, пока другой пишет ИИ для `HealerBehavior` — без конфликтов в файлах. |
| **Планируемое расширение** | Если в будущем планируется система классов, прокачка или смена режима боя — паттерн уже подготовлен для гибкости. |

**Заключение**: Паттерн «Стратегия» превращает вариативное поведение из проблемы в настраиваемый компонент. Для игры с потенциальным развитием и множеством типов юнитов это инвестиция в поддерживаемость кода, которая окупается при первом же добавлении нового класса персонажей.
